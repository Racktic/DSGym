# Dev Log 2026-03-22: DSPredict 数据扩展与整合

## 一、目标

将 MLE-Dojo 的竞赛任务转换为 DSPredict 格式，并通过 target-swap 技术扩展 tabular 任务数量，统一所有数据的格式和 Dataloader。

## 二、完成的工作

### 1. MLE-Dojo → DSPredict 格式转换（mle_dojo split）

**来源**：MLE-Dojo 项目中的 125 个 Kaggle 竞赛（排除已存在于 DSPredict easy/hard 中的 8 个）

**流程**：
1. 从 `MLE-Dojo/prepare/mle.json` 读取竞赛元数据（title, tags, host, participation）
2. 从 `MLE-Dojo/mledojo/competitions/{name}/info/description.txt` 读取竞赛描述
3. 解析 description.txt 的分段结构（Description, Evaluation, Submission File, Dataset Description）
4. 转换为 DSPredict 标准 JSON 格式（7 个字段）
5. 通过 Kaggle API 下载竞赛数据到 `data/data/dspredict-mledojo/`
6. 用实际文件列表和 CSV 列名信息丰富 dataset_description（LLM 辅助润色 12 个）
7. 移除无法下载的任务（3 个规则未接受 + 2 个超大 + 1 个缺标签）

**脚本**：
- `convert_mle_to_dspredict.py`：JSON 格式转换
- `download_mle_data_v2.py`：带限速重试的批量下载
- `scripts/enrich_mledojo_tasks.py`：用实际数据文件信息丰富描述

**结果**：111 个任务，涵盖 Tabular、CV、NLP 等类型

**下载过程中的问题**：
- Kaggle API 限速（429 Too Many Requests）：每 5-7 个请求触发一次，通过 30s 间隔 + 10min 退避解决
- 所有竞赛需先在 Kaggle 网站手动接受规则（403 Forbidden）
- 3 个竞赛无法接受规则（已关闭）
- 2 个超大数据集（herbarium-2022-fgvc9 149GB, vinbigdata 类似）超时，直接移除
- 部分数据以 .gz/.7z/.tar.gz 格式下载，需额外解压

### 2. Target-Swap 数据扩展（swap split）

**思路**：复用 dspredict-easy 的 tabular CSV 数据，更换预测目标列生成新任务

**流程**：
1. 分析 48 个 tabular 数据集的列结构，筛选候选目标列
2. 筛选条件：排除 ID 列、常量列、高缺失列、高基数字符串列、特征泄漏列
3. 每个数据集选最多 3 个候选，按质量分排序
4. 重新划分 train/test（80/20），生成 sample_submission.csv 和 ground_truth.csv
5. 可学习性检查（DecisionTree vs DummyClassifier/DummyRegressor）
6. 自动选择评估指标（RMSE/RMSLE/AUC/LogLoss/Accuracy）
7. LLM（Together AI, Qwen 235B）润色任务描述

**脚本**：`scripts/generate_swap_tasks.py`

**结果**：67 个新任务（49 回归 + 18 分类），来自 30 个原始数据集

### 3. Hard Split 数据下载

将 DSGym 原有的 hard.json 对应的 54 个竞赛数据全部下载到 `data/data/dspredict-hard/`。

**结果**：54/54 全部成功

### 4. Dataloader 统一

修改 `dsgym/datasets/loaders/dspredict.py`，支持 5 个 split：

```python
SPLIT_CONFIG = {
    "easy":     ("easy.json",     "dspredict-easy"),
    "hard":     ("hard.json",     "dspredict-hard"),

sample["extra_info"]["data_files"] = {"virtual": [...], "local": [...]}
sample["extra_info"]["metadata"] = {...}  # 新增，包含 task_type, metric 等
```

新增 `SwapSubmissionMetric`（`dsgym/eval/metrics/dspredict/swap_metric.py`）：

- swap split 加载时自动使用此 metric

## 三、MLE-Bench 数据泄漏排查

MLE-Dojo 的 200 个任务中有大量和 MLE-Bench 原题重叠（共用相同 Kaggle 竞赛 slug = 完全相同的数据）。如果用于训练，评测时会数据泄漏。

**排查结果**：
- mle_dojo split 与 MLE-Bench 重叠 **51 个** → 全部踢掉
- hard split 与 MLE-Bench 重叠 **6 个** → 标记，训练时需注意排除
- easy / swap 与 MLE-Bench 重叠 **0 个** → 安全

踢掉后 mle_dojo 从 111 → **60 个**。另删除 3 个因其他原因不可用的任务（2 个超大 + 1 个缺标签），这 3 个恰好也在 MLE-Bench 重叠列表中。

## 七、数据验证与修复（2026-03-23）

### Hard Split 重新下载

之前的 hard 数据是用自定义脚本（`download_mle_data_v2.py` + kaggle CLI）下载的，发现大量文件路径与 `hard.json` 中 description 不一致。重新使用 DSGym 官方 `KaggleChallengeDownloader`（`kagglesdk` + `zipfile.extractall`）下载全部 54 个竞赛数据。

**结论**：路径不一致问题与下载方式无关，是 Kaggle 数据本身的 zip 内部目录结构与竞赛描述不一致导致的。

### Hard Split 数据问题修复

详见 `dev_logs/hard_split_data_issues.md`，总结如下：

| 目录双层嵌套 | 1 | `ln -s` 消除嵌套 |

### Hard 全集推理结果（AIDE baseline, 54 tasks）

| 类别 | 数量 | 占比 |
|---|---|---|
| 有 Kaggle 分数 | 16 | 29.6% |
| 提交但被 Kaggle 拒绝 | 12 | 22.2% |
| 未产出 submission | 26 | 48.1% |

有分数的 16 个任务：Mean percentile 21.89，Median 20.11，仅 1 个 above 50%（home-data-for-ml-course, 91.66%）。

**主要失败原因：**
- Output token 截断（25/26 未产出 submission 的任务受影响）：hard 任务代码复杂，AIDE 自包含模式每轮写完整脚本，超出 Together AI max_output_tokens 限制
- 提交格式被 Kaggle 拒绝：5 个 schema 不匹配，6 个竞赛已关闭不接受晚提交，1 个预测值为空
- 缺少 Python 包：~9 个 CV/NLP 任务

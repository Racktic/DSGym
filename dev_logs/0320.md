# Dev Log 2026-03-20: Cross-Task Memory 模块设计、实现与实验

## 一、背景与动机

### 从 AIDE vs EET 分析出发

在之前的 action space 对比实验中，我们发现：

| 实验 | Mean Pub Percentile | Median | Above Median |
|---|---|---|---|
| AIDE (t20, d5) | 45.94 | 47.62 | 45% |
| EE no-terminate | 40.73 | 36.50 | 30% |
| EET v3 | 39.05 | 35.62 | 25% |

**AIDE 显著优于 EET**，核心原因有三：
1. **每轮自包含代码**：免疫状态丢失，错误不传播（EET 的 3/5 差距最大 case 都因级联失败）
2. **硬编码 Draft→Improve 流程**：强制模型多样性 + 系统化单变量调优
3. **失败即丢弃，从 best_node 重新改**：EET 出错后只能在 broken state 上修

### 提出 Cross-Task Memory

所有方法的共同瓶颈：**每个 task 从零开始探索**。如果能跨 task 积累经验（如"CatBoost 在回归任务上通常比 XGBoost 好"、"MedAE 指标用 `reg:absoluteerror` 作为 objective"），可以减少无效探索。

## 二、AIDE 代码结构分析

### AIDE 每轮代码差异

对 AIDE v1 的 299 对连续 improve turn 的统计：
- 平均 **78.9%** 行完全相同（中位数 86.8%）
- ~45% 的 turn 对只改 1-2 个超参数
- ~30% 加/删一个 feature engineering 块
- 数据加载 boilerplate 每轮几乎一样（前 20 行 90.3% 相同）

### AIDE Debug 成功率
- 严格成功率：**54.1%**（33/61）
- TimeoutError 修复率 65%，TypeError 修复率仅 35%
- Debug chain 最长 2 次，失败后自动回到 improve（从 best_node 重新改）

### AIDE 的 4 个 Action
1. **Draft**（前 5 轮强制）— 从零写新方案
2. **Improve**（后续默认）— 基于最佳方案做一个原子改动
3. **Debug**（上轮 buggy 时 50% 概率）— 修复 bug
4. **Final Submission**（最后一轮）— 用最佳方案生成提交

## 三、Memory V1 实现

### 设计
- **存储**：`CrossTaskMemory` 类，JSON 文件持久化
- **写入**：每轮执行完后自动提取 challenge_name、plan、model_type（正则检测）、score、buggy、insight
- **检索**：排除当前 task，按 relevance 排序（成功改进 > 正常运行 > buggy），取 top 15
- **注入**：格式化为文本块，附加到每轮 prompt 的 memory section

### 集成
- AIDE：每轮 `_build_turn_instruction` 中注入 cross-task memory
- EET：注入到 system prompt

### CLI 支持
- 新增 `--memory-path` 参数传入 memory JSON 文件路径

## 五、V1 问题诊断

### Bug 1: 并发写入冲突
8 个 worker 并行跑，同时读写同一个 JSON 文件：
- 没有文件锁，多个 worker 同时写会互相覆盖
- 如果用多进程，每个 worker 有自己的 memory 副本，互相看不到对方写入的内容
- 极端情况下两个同时写可能导致 JSON 文件损坏

### Bug 2: 每轮都写，噪声太多
每轮都调用 `store_from_turn()`，导致：
- AIDE 621 条（38 task × ~16 条/task），大量是无 score 提升的 improve turn
- EET 478 条，其中 63% 没有 score
- 高质量信号（如"发现 objective function 对齐"）被淹没在噪声中

### Bug 3: 检索策略太粗糙
- 只按 "有 score 改进 > 有 score > buggy" 排序
- 没有按 task 类型/相似度过滤
- Top 15 可能全来自同一类型的 task，其他类型的经验完全丢失

## 六、Memory V2 改进

### 改进 1: 选择性写入（只写高价值事件）

**AIDE 触发条件**：
1. 刷新最佳分数 → `store_improvement()`
2. Debug 成功修复 bug → `store_debug_fix()`
3. Task 结束 → `store_task_summary()`

**EET 触发条件**：
1. 刷新最佳分数 → `store_improvement()`
2. Task 结束 → `store_task_summary()`

预计条目数：每个 task 3-5 条（vs V1 的 ~16 条），总共 ~150 条（vs V1 的 621 条）

### 改进 2: 文件锁解决并发

```python
def _append_entry(self, entry):
    lock_path = self.memory_path + ".lock"
    with open(lock_path, "w") as lock_file:
        fcntl.flock(lock_file, fcntl.LOCK_EX)  # 排他锁
        try:
            existing = load_from_disk()
            existing.append(entry.to_dict())
            atomic_write(existing)  # os.replace 原子替换
        finally:
            fcntl.flock(lock_file, fcntl.LOCK_UN)
```

- 写入用排他锁（LOCK_EX）
- 读取用共享锁（LOCK_SH）
- 不缓存，每次从磁盘读最新
- 写入通过临时文件 + `os.replace` 实现原子性

### 改进 3: 检索优先级随当前 action 变化

优先级不再固定，而是根据当前 agent 正在执行的 action 动态调整：

| 当前 action | 优先级排序（分数越高越靠前） |
|---|---|
| draft / explore | task_summary (3.0) > improvement (1.5) > debug_fix (1.0) |
| improve / exploit | improvement (3.0) > task_summary (1.5) > debug_fix (1.0) |
| debug | debug_fix (3.0) > improvement (1.5) > task_summary (1.0) |

逻辑：
- **Draft 阶段**需要知道"什么模型/方案在类似 task 上效果好" → summary 优先
- **Improve 阶段**需要知道"什么具体改动能提升分数" → improvement 优先
- **Debug 阶段**需要知道"这类 bug 怎么修、怎么避免" → debug_fix 优先

分数是硬排序，不是加权混合（所有 3.0 的条目排在所有 1.5 的前面）。

AIDE 每轮重新构建 prompt，所以每轮都会根据当前 action 重新检索。EET 在 task 开始时注入一次 system prompt，用默认的 draft 优先级。

### 改进 4: max_per_task 保证多样性

检索时每个 task 最多取 3 条（`max_per_task=3`），防止某个 task 的大量条目占满 top-k。
比如 top_k=15、max_per_task=3 → 至少覆盖 5 个不同 task 的经验。

### 改进 5: Insight 质量提升

V1 的 insight 是机械拼接（score + plan 原文），V2 改为精炼总结：

**improvement insight**:
```
Score: 0.85 -> 0.82 (3.5% improvement). Method: LightGBM. Key change: Switch to LightGBM with target encoding for categorical features
```
包含：变化幅度、用了什么模型、关键改动是什么。

**debug_fix insight**:
```
Bug: TypeError: got an unexpected keyword argument 'multi_class'. Fix: Removed multi_class param. Avoid: Check API parameter names match the library version
```
包含：bug 是什么、怎么修的、下次怎么避免。

**task_summary insight**:
```
Best model: CatBoost. Score: 0.91 (3.4% improvement over baseline). Strategy: CatBoost with ordinal encoding
```
包含：最终用了什么模型、效果如何、核心策略。

辅助函数：
- `_summarize_plan()`: 从模型的 goal 文本中提取核心信息（去掉 filler，取第一句，限制长度）
- `_error_avoidance_tip()`: 根据 error type 生成标准化的避坑建议

## 八、新实验：EET Self-Contained 与 AIDE Memory V3

### 实验目的

1. **EET self-contained**：验证"EET 差是因为错误传播/状态丢失，还是因为 action 设计本身不行"
2. **AIDE memory V3（draft 不注入 memory）**：验证"memory 的负面影响是否主要来自 draft 阶段的多样性被压缩"

### 实现方式（纯增量，不修改原有代码）

**EET self-contained**：
- 新增 `EET_SYSTEM_PROMPT_SELF_CONTAINED`（prompt 变体），将 "Code execution is continuous — variables persist across steps" 替换为 "Your code must be SELF-CONTAINED every step"
- 新增 CLI 参数 `--self-contained` + EETAgent 的 `self_contained` kwarg
- 原有 prompt 完全不变

**AIDE memory V3**：
- 新增 CLI 参数 `--no-draft-memory` + AIDEAgent 的 `no_draft_memory` kwarg
- 在 `_build_turn_instruction` 中，当 `action == "draft"` 且 `no_draft_memory=True` 时跳过 cross-task memory 注入
- Improve/Debug 阶段仍然注入 memory

### 全局排名

| 排名 | 实验 | Mean % | Median % | Above Median | 失败数 |
|---|---|---|---|---|---|
| 1 | **AIDE+mem V2** | **48.83** | **52.61** | **18/38** | 6 |
| 2 | AIDE baseline | 45.94 | 47.62 | 13/38 | 9 |
| 3 | **EET self-contained** | **44.44** | **46.14** | 12/38 | 7 |
| 4 | **AIDE+mem V3** | 43.76 | 45.26 | 13/38 | **3** |
| 5 | EE no-terminate | 40.73 | 36.58 | 9/38 | 8 |
| 6 | EET v3 | 39.05 | 36.29 | 8/38 | 6 |

### EET Self-Contained 结论

**自包含代码确实有效。** EET self-contained 比原版 EET v3 提升 **+5.4 mean percentile**（44.44 vs 39.05），win rate 16:9。

但 EET self-contained 仍略低于 AIDE baseline（44.44 vs 45.94，差 -1.5）。说明：
- 错误传播/状态丢失是 EET 差的**主要原因**（修复后差距从 -6.9 缩小到 -1.5）
- AIDE 的 Draft→Improve 硬编码流程仍有微弱优势

### AIDE Memory V3 结论

V3 在 V2 的 5 个大输 task 上的表现：

| Task | V2 vs Base | V3 vs Base | 修复了吗 |
|---|---|---|---|
| s3e19 | -32.3 | **+14.1** | ✅ 完全修复 |
| s3e14 | -32.0 | -7.8 | ✅ 大幅改善 |
| s4e2 | -25.9 | 0.0 | ✅ 完全修复 |
| s3e24 | -21.6 | -12.0 | 部分改善 |
| s3e25 | -21.4 | -35.8 | ❌ 更差了 |

**3/5 修复，1 改善，1 更差。** 证实了 draft 阶段不注入 memory 能减少 CatBoost 锁定等问题。

但 V3 整体不如 V2（43.76 vs 48.83）——**V3 减少了大输但也减少了大赢**。V3 的优势是**最低失败率**（3/38）。

### V2 大输的根因分析

对 V2 的 5 个大输 task 做了逐个 trajectory 分析，发现 memory 导致三种失败模式：

1. **CatBoost 锁定**（s3e24, s4e2, s3e14）：memory 中 CatBoost 成功经验太多，Draft 阶段就被引导到 CatBoost，后续 Improve 只能在 CatBoost 上微调，错过了 XGBoost/LightGBM 等 task-specific 最优方案
2. **有毒 memory 传播**（s3e19 → s3e24）：s3e19 因 lag feature 数据泄露获得虚假高分（CV=5.19 但 Kaggle=110），这个 "88.6% improvement" 被写入 memory，误导后续 task
3. **Token 预算被挤占**（s3e19）：cross-task memory + 自身 history 太长，导致代码被截断（15/20 轮没有 `</python>` 闭合标签）

### 稳定性 vs 性能的 Tradeoff

- **V2**：高方差，大赢大输，均值最高（48.83）
- **V3**：低方差，减少大输但没有大赢，均值较低（43.76）
- 最优方案可能在 V2 和 V3 之间

### 讨论：如何解决 Memory 导致的过早 Exploit

Memory 的核心问题是 **explore-exploit 的 exploit 偏差**——前几个 task 的成功经验被过度放大，导致后续 task 全都先试同一种方案。

讨论了几个方案：
1. **Draft 阶段不注入 memory**（V3 已验证，有效但牺牲性能）
2. **动态统计模型使用频率**，在 prompt 里提示 agent 探索低频方法
3. **UCB 风格的多样性奖励**（理论方案）
4. **Thompson Sampling 思路**（理论方案）

## 八.5、新实验：Best-Node 策略与频率统计（2026-03-24）

### 代码改动

**Best-node 策略**（`--best-node-strategy best`）：
- 原有行为（latest）：Improve 时取 `good_nodes[-1]`（最近一个成功方案）
- 新行为（best）：Improve 时取 score 匹配 `best_score` 的 node（更接近原版 AIDE 树搜索）

**Good nodes 分数修正**：
- 原有 bug：`good_nodes` 存的 score 是 `best_score`（系统级历史最佳），导致所有 node 分数一样
- 修正：从 execution output 提取当前轮的实际 validation score（正则匹配 `Validation Score: xxx` 或 `CV Score: xxx`，98.2% 匹配率）

**频率统计**（直接改，非参数）：
- 在 `format_for_prompt` 输出的 memory 末尾附加模型使用频率统计
- 示例：`Model usage frequency across tasks: CatBoost (15x), XGBoost (8x), LightGBM (3x). Consider exploring under-represented approaches for diversity.`

### 实验结果

| 实验 | Mean % | Median % | Above Median | 提交成功率 |
|---|---|---|---|---|
| AIDE baseline | 45.94 | 47.62 | 45% | 76% |
| **V2 latest-node** | **48.83** | **52.32** | **56%** | **84%** |
| V2 best-node | 45.11 | 45.77 | 41% | 84% |
| V2 best-node + freq | 42.84 | 42.80 | 36% | 66% |

### Best-node vs V2 latest-node

**Best-node 有害**：均值 -3.7，win/loss 13:15。s4e2 从 40.72 暴跌到 5.43。

原因分析：`best_score`（模型自报的系统级最佳）和 `good_nodes` 的 `current_score`（从 execution output 提取）之间匹配不上，导致 `_get_best_approach_summary` 找不到正确的 node，fallback 到最后一个。

### Freq + best-node vs V2

**更差**：均值 -5.1，提交成功率从 84% 暴跌到 66%。

唯一亮点：**s3e19 从 0.85 飙升到 69.08**（freq 提示引导模型尝试了不同方法），但其他 task 全面退步。提交成功率下降可能是因为 freq 统计增加了 prompt 长度，加剧了 output token 截断。

### V2 的 5 个大输 task 追踪

| Task | Baseline | V2 | Best-node | Freq |
|---|---|---|---|---|
| s3e19 | 33.1 | 0.9 | 8.6 | **69.1** ✅ |
| s3e14 | 62.0 | 30.0 | 33.6 | 失败 |
| s4e2 | 66.6 | 40.7 | **5.4** ❌ | 50.5 |
| s3e24 | 71.7 | 50.1 | 52.9 | 失败 |
| s3e25 | 55.3 | 33.9 | 24.9 | 30.6 |

### 结论

**V2 latest-node 仍然是最佳配置。**

- Best-node 策略的失败说明模型自报的 best_score 和 good_nodes 里的实际分数之间存在不一致，简单的分数匹配不可靠
- Freq 统计的构想是对的（s3e19 大幅修复），但副作用太大（prompt 变长 → 截断增加 → 成功率下降）
- 核心问题未解决：如何在不增加 prompt 长度的前提下引导 draft 多样性
